# LDD 기반 웹소설 창작 시스템 스펙 시트 v1.1

---

## 1. 프로젝트 정의

### 1.1 개요

- **프로젝트명**: 이세계 커뮤니티 시뮬레이션 기반 웹소설 제작 시스템
- **슬로건**: "나는 이세계의 김유식이 된다"
- **목적**: 판타지 아카데미 세계관에서 커뮤니티 시뮬레이션을 통해 작가가 예측 불가능한 재미 요소를 발견하고, 이를 바탕으로 장편 웹소설을 제작하는 시스템

### 1.2 LDD (LLM-Driven Development) 정의

> 코드로 로직을 구현하는 대신, 자연어 문서로 행동 규칙을 정의하고 LLM이 해석하여 실행하는 개발 패러다임. 엔지니어링의 핵심은 로직 설계가 아니라 **컨텍스트 관리**.

| | 전통적 개발 | AI-Assisted | LDD |
|---|---|---|---|
| 로직 | 코드 | 코드 (AI 보조) | 자연어 문서 |
| 런타임 | 컴파일러 | 컴파일러 | LLM 자체 |
| 변경 방식 | 코드 수정 → 배포 | 코드 수정 → 배포 | 텍스트 파일 수정 |
| LLM 역할 | 없음 | 코드 생성 도구 | 실행 엔진 |

**이 시스템에서:**
- "프로그램" = 자연어 문서 집합 (master_map, playbook, character cards 등)
- "프로세서" = LLM (오케스트레이터 + 서브 에이전트 + 페르소나 인스턴스)
- "하드웨어" = 2층 실행 인프라 (오케스트레이션 레이어 + API 시뮬레이션 레이어)

## 2. 설계 원칙

### 2.1 고정 vs 유동

**고정 (엔진)**:
- 오케스트레이션 레이어 (클로드 코드/코덱스 — 파일 I/O, 작가 대화, 문서 관리)
- 시뮬레이션 실행 레이어 (API 직접 호출 — 라운드 기반 실행, 정보 격리, 병렬 페르소나 실행)

**유동 (콘텐츠)**: 세계관, 커뮤니티 구조, 캐릭터, 시뮬레이션 기억, simulation_playbook, style_bible — 전부 자연어 문서로 관리되며 작품 진행과 함께 변화

### 2.2 컨텍스트 관리 원칙

"경량 인덱스로 어디를 볼지 판단 → 필요한 부분만 로딩"

- 오케스트레이터는 항상 master_map.md만 상시 보유
- 작업 유형에 따라 필요한 하위 인덱스만 추가 로딩
- 무거운 읽기/수정 작업은 서브 에이전트에게 위임
- LLM의 자연어 이해 능력을 신뢰하고, 구조는 "책의 목차" 수준으로 최소화

### 2.3 실행 아키텍처: 2층 분리

> **MVP 교훈**: 클로드 코드/코덱스의 서브 에이전트는 시스템 프롬프트 수준에서 "소프트웨어 엔지니어링 어시스턴트"로 정의되어 있어, 판타지 세계의 페르소나로 역할극을 수행할 때 개발자처럼 행동하는 문제가 발생함. 프롬프트 엔지니어링만으로는 안정적 해결 불가.

**해법: 역할에 따른 실행 환경 분리**

```
[오케스트레이션 레이어] — 클로드 코드/코덱스
  · 마스터 맵 관리, 파일 I/O, 작가와의 대화
  · 시뮬레이션 기획 (반응자 선별, 컨텍스트 패키지 준비)
  · 결과 수합 → Writer Report 생성 → 상태 갱신
  · 세계관 수정의 서브 에이전트 디스패치
  → "소프트웨어 어시스턴트" 역할이 오히려 적합한 작업

[시뮬레이션 실행 레이어] — LLM API 직접 호출
  · 순정 모델에 페르소나 시스템 프롬프트만 주입
  · 경쟁하는 시스템 지시 없음 → 안정적 역할극 수행
  · 각 API 호출이 물리적으로 독립된 세션 → 정보 격리 구조적 보장
  · temperature, top_p 등 파라미터 자유 제어
  · Batch API 활용 시 비용 50% 절감 가능
```

**API 호출의 이점:**

| 항목 | 클로드 코드 서브 에이전트 | API 직접 호출 |
|---|---|---|
| 시스템 프롬프트 | 코딩 어시스턴트 고정 | **완전 제어** (페르소나만 주입) |
| 정보 격리 | 컨텍스트 공유 여지 있음 | **물리적 세션 분리** |
| 파라미터 제어 | 도구 제한 내 | temperature/top_p 자유 조절 |
| 비용 최적화 | 불가 | Batch API 활용 가능 |
| 역할극 안정성 | 불안정 (개발자 회귀) | **높음** (경쟁 지시 없음) |

**작업별 실행 레이어 배정:**

| 작업 | 실행 레이어 | 이유 |
|---|---|---|
| 세계관 수정 (탐색/수정/검증) | 오케스트레이션 | 파일 읽기/쓰기 + 분석 작업 |
| 집필 기획 대화 | 오케스트레이션 | 작가와의 대화 + 문서 참조 |
| 시뮬레이션 페르소나 실행 | **API 직접 호출** | 역할극 — 순정 모델 필수 |
| 스타일 분석 | 오케스트레이션 | 문서 비교 + 패턴 추출 |
| 상태 갱신 | 오케스트레이션 | 파일 쓰기 + 작가 대화 |
| 온디맨드 캐릭터 생성 | API 직접 호출 또는 오케스트레이션 | 대량 배치 시 API가 효율적 |

### 2.4 작가 승인 원칙

모든 "정식 반영" 작업의 공통 인터페이스:

```
LLM 판단/제안 → 작가 승인/수정/거부 → 합의된 내용만 반영
```

적용 범위: 세계관 수정, 시뮬레이션 상태 갱신, 집필 기획, 스타일 피드백, 커뮤니티 생성, 캐릭터 레귤러 선정 등 모든 정식 반영 작업

## 4. 세계관 관리 시스템

### 4.1 월드 바이블 구조

**world_bible_index.md**
- 각 섹션의 내용 요약
- 주요 개체 크로스레퍼런스 (개체명 → 관련 섹션 목록)

**섹션 파일들** (`world_bible/` 폴더)
- 목차/섹션 수준으로 분리된 자연어 문서
- 완전한 정형 스키마 불필요

### 4.2 세계관 수정 4-Phase 워크플로우

```
[Phase 1: 아이디어 대화]
작가 ↔ 오케스트레이터 (master_map + world_bible_index 참조)
  · 서브 에이전트 미투입, 가볍고 빠른 대화
  · 변경안 구체화

        ↓ 변경안 확정 시

[Phase 2: 충돌 탐색]
오케스트레이터 → 크로스레퍼런스 참조 → 관련 섹션 식별
  → 서브 에이전트들에게 병렬 읽기 지시
  → 서브 에이전트: 관련 내용 + 충돌 가능성 보고 (변경 요약 수준)
  → 오케스트레이터: 보고 종합 → 충돌 여부 작가에게 제시

충돌 분기:
  ├─ 경미한 충돌 (서술 수준) → 오케스트레이터가 즉석 해결안 제시 → Phase 3
  └─ 대규모 충돌 (연쇄 설정 변경) → 아래 대규모 변경 프로세스

        ↓ 해결 방안 확정

[Phase 3: 변경 적용]
오케스트레이터 → 서브 에이전트들에게 수정 지시
  · 서브 에이전트: 각 섹션 수정 후 변경 요약 보고
  · 오케스트레이터: 모든 보고 수령 확인

        ↓ 보고 완료 후

[Phase 4: 마스터 맵 갱신]
오케스트레이터: world_bible_index 업데이트
  · 섹션 내용 요약 갱신
  · 크로스레퍼런스 갱신
  · 변경 이력 누적 (changelog 역할)
```

### 4.3 대규모 변경 프로세스

```
1. 오케스트레이터가 대규모 충돌 판정
2. 작가 ↔ 오케스트레이터: 고급 추론 모델에 넣을 프롬프트를 대화로 합의
3. 프롬프트 로그 저장
4. 작가가 프롬프트 + 관련 문서를 고급 추론 모델에 수동 제출
5. 재작성된 문서를 시스템에 주입
6. 오케스트레이터 → 서브 에이전트에게 3자 대조 검증 지시
   입력: 프롬프트 로그(의도) / 이전 문서(before) / 새 문서(after)
   검증:
     · 의도된 변경이 실제로 반영되었는가?
     · 의도하지 않은 부분이 바뀌지 않았는가?
     · 새로 도입된 내용이 다른 섹션과 충돌하지 않는가?
7. 서브 에이전트 검증 보고
8. 오케스트레이터가 마스터 맵 갱신
```

## 5. 캐릭터 시스템

> **v1.2 전면 재설계**: 사전 조형 방식(200개 카드 선제작)을 폐기하고, 심리 좌표 기반 모집단 + 온디맨드 생성 방식으로 전환. 변경 배경은 `docs/design/character_population_methodology.md` 참조.

### 5.1 캐릭터 ID 체계 (2계층)

**[P-* 모집단 슬롯]** — 심리 좌표만 존재, 인물 미조형
- 범위: P-0001 ~ P-3600
- 저장 내용: 빅5 수치(O/C/E/A/N) + 파생 지표(DT/NFC/SM) + status
- status: `uninstantiated` → `instantiated` → `named`
- 온라인 페르소나: 사전 작성 금지. 시뮬레이션 결과에서 귀납적으로 확정

**[CH-* 명명 캐릭터]** — 시뮬/집필에서 발굴된 인물
- 소환 시 P-* 슬롯에서 즉석 조형 후 CH-* ID 발급
- 풀 카드 구조(정체성 레이어 + 온라인 성향 레이어)는 CH-* 단계에서만 작성

**캐릭터 카드 구조 (CH-* 전용)**

[정체성 레이어]
- 기본 정보 (학년, 탑, 출신, 소속)
- 빅5 수치 (P-* 슬롯에서 상속)
- 오프라인 코어 (핵심 대사/성향)
- 욕망 / 공포 / 결함
- 비밀 (정보 레벨)
- 핵심 관계

[온라인 성향 레이어] — 시뮬레이션 후 귀납 확정
- 활동 주제 성향
- 익명 선호도
- 온라인 말투 스타일
- 회피 주제/상황
- 활동 강도

### 5.2 캐릭터 Tier 진행

```
P-* 슬롯 (uninstantiated)
  · 심리 좌표만 존재
  · 시뮬레이션 반응자 선별 풀

        ↓ 시뮬레이션 소환 시

instantiated
  · 오케스트레이터가 즉석 조형 (이름 + 코어 동기 + 탑/학년)
  · 시뮬레이션 실행 → 반응 생성
  · 흥미롭지 않으면 → 결과만 보관, 슬롯 재사용 가능

        ↓ Writer Report에서 반복 주목 + 작가 승인

named (CH-* 발급)
  · 풀 카드 작성
  · 온라인 성향 레이어 시뮬 결과 기반 확정
  · persona_state.md 개설
  · Voice Pack으로 확장
```

| status | 설명 |
|---|---|
| uninstantiated | 심리 좌표만 존재, 미소환 |
| instantiated | 시뮬에서 소환·조형됨, 이름 있음 |
| named | 레귤러 선정 완료, CH-* 풀 카드 보유 |

### 5.3 캐릭터 인덱스 (character_index.md)

좌표 점유 레지스트리 — 중복 방지와 반응자 선별에 사용:

```
[점유 좌표 레지스트리]
탑 × 학년 × 배경유형 × 동기아키타입 조합 → 점유 여부

[반응자 선별용 역색인] (instantiated 이상만)
빅5 구간별:  NFC 상위 20% → [ID 목록]
DT 구간별:  DT 상위 10% → [ID 목록]
탑별:        흑탑 → [ID 목록]
```

새 캐릭터 생성 시 카드 대 카드 비교 불필요 — 레지스트리에서 빈 좌표 탐색 후 할당.

### 5.4 온디맨드 생성 흐름

```
시드 입력
  ↓
오케스트레이터: population_grammar 참조
  → 반응 예상 심리 프로파일 추론
  → character_index에서 해당 좌표 구간 탐색
  ↓
미점유 P-* 슬롯 선택 → 즉석 조형
  ↓
시뮬레이션 실행
  ↓
결과 평가:
  흥미로운 반응 → instantiated 유지, Writer Report 기록
  평범한 반응  → 결과 보관, 슬롯 초기화 가능
```

## 6. 커뮤니티 시뮬레이션 시스템

### 6.1 시뮬레이션 모드

**[탐색 모드]** — 집필 전, 선택적
- 목적: "이 사건을 쓰면 커뮤니티가 어떻게 반응할까?" 확인, 집필 방향 결정에 활용
- 시드 출처: 작가의 가설
- 결과: Writer Report만 출력 → **상태 갱신 없음**
- 복수 시드 후보 비교 실험 가능

**[정식 모드]** — 집필 후, 선택적
- 목적: 확정된 사건에 대한 커뮤니티 반응 기록
- 시드 출처: 확정 원고에서 오케스트레이터가 추출 + 작가 확인
- 결과: Writer Report + **상태 갱신 (작가 승인)**

**시뮬레이션 실행 여부 판단 기준** (simulation_playbook에 포함):

| 적합 | 불필요 |
|---|---|
| 공개 사건 | 비공개 개인 서사 |
| 논란/갈등 유발 가능성 있음 | 캐릭터 내면 묘사 |
| 커뮤니티에 알려진 인물 관련 | 커뮤니티 미인지 사건 |
| 여러 세력/진영에 영향 | 소소한 일상 묘사 |

판단: 오케스트레이터 제안 → 작가 최종 결정

### 6.2 시드 정의 및 정제 프로세스

**형태**: 자연어 자유 기술 + 오케스트레이터의 최소 필수 항목 정제

**오케스트레이터 체크리스트** (빠진 것만 질문):
1. **공개 범위** — 전체 공개 / 특정 집단만 / 소문 수준 (맥락에서 명확하면 생략)
2. **관련 핵심 인물/집단** — 사건 설명에 포함되면 생략
3. **시뮬레이션 모드** — 집필 전/후로 대부분 자동 판단, 애매한 경우만 확인

**시드 구성:**
```
탐색 시드: 사건 요약(자연어) + 공개 범위 + 관련 인물(선택)
정식 시드: 위 + 발생 회차 번호
```

### 6.3 반응자 선별 퍼널

```
전체 캐릭터 풀
  ▼ [1단계: 관련성 필터링] — character_index 기반 (카드 직접 읽기 불필요)
  ~300~500명: 사건을 "인지"하는 캐릭터
  ▼ [2단계: 반응 강도 분류]
  ├─ 무반응/눈팅 (~300명)   → 처리 없음, "인지"만 기록
  ├─ 소극적 반응 (~100명)   → 추천/비추천만, 그룹 일괄 처리
  ├─ 댓글 작성 (~30~50명)   → 1~3줄 개별 생성
  └─ 글 작성/핵심 반응 (~5~15명) → 풀 시뮬레이션 (Voice Pack 포함)
```

### 6.4 시뮬레이션 턴 루프

**핵심 원칙**: 같은 라운드 내 페르소나들은 서로의 출력을 볼 수 없음 (비동기 정보 격리)
**실행 환경**: 페르소나 실행은 **LLM API 직접 호출** (Section 2.3 참조) — 순정 모델에 페르소나 시스템 프롬프트만 주입, 각 호출이 물리적으로 독립된 세션이므로 정보 격리가 구조적으로 보장됨

```
[Phase 0: 시드 주입 + 준비] — 오케스트레이션 레이어
오케스트레이터: master_map → 하위 맵 로딩 → 반응자 선별 퍼널 실행
  → 참여 페르소나별 컨텍스트 패키지 조립
  → API 호출 배치 준비
출력: 관련 게시판 목록, 활성 페르소나 풀, 예상 라운드 수

[Phase 1: 라운드 루프] — API 시뮬레이션 레이어 (반복, 일반 3~5R / 대형 5~7R)
  R1 퍼스트 무버: 2~5명 최초 글 작성 (병렬 API 호출, 물리적 세션 분리)
    → 글 "게시" (이후 라운드에서 공개)
  R2 초기 반응: 댓글 작성자 + 소극적 반응 일괄 처리
    → 댓글 + 추천/비추천 "게시"
  R3 토론 심화:
    · 댓글 간 충돌 → 대댓글
    · 교차 게시판 유입
    · 신규 진입자 (뒤늦게 참여)
    · 반전 가능성 탐색 (상태 카드 참조, 예상과 반대 행동 캐릭터)
  R4~: 수렴 감지 시 종료

  매 라운드 간:
    오케스트레이터가 결과 수합 → board_state 업데이트 → 다음 라운드 API 호출 준비
    수렴 판정: 새 논점 없이 반복 / 최대 라운드 도달

[Phase 2: 상태 갱신] — 오케스트레이션 레이어, 정식 모드만, 작가 승인 후 실행
  참여 페르소나 상태 카드 갱신
  게시판 상태 갱신
  커뮤니티 기억 갱신 (큰 사건인 경우)
  캐릭터 인덱스 갱신

[Phase 3: Writer Report 생성] — 오케스트레이션 레이어
  1. 핵심 갈등 구조
  2. 주목할 장면 후보
  3. 대사/텍스트 후보 (시뮬레이션에서 생성된 실제 글)
  4. 다음 회차 시드 후보
  5. 수치 요약 (참여자 수, 게시글/댓글 수, 추천/비추천, 가장 뜨거운 스레드)
```

### 6.5 API 호출 컨텍스트 조립

**1건의 API 호출 = 1명의 페르소나가 1개의 글/댓글을 작성**

```
system_prompt (페르소나 정의 — 경쟁 지시 없는 순정 모델):
  · 캐릭터 카드 (정체성 레이어 + 온라인 성향 레이어)
  · persona_state (현재 입장/관계/감정 — 레귤러인 경우)
  · 해당 게시판 layer_b (문화/톤/금기)
  · 역할 지시: "당신은 [캐릭터명]입니다. 아래 게시판에서 활동하는
    이용자로서 반응하세요."

user_message (상황 정보 — 매 라운드 갱신):
  · community_memory (공유 기억)
  · board_state (이전 라운드까지의 글 — 같은 라운드 글 미포함)
  · 시드 이벤트
  · 반응 형식 지시 (글 작성 / 댓글 / 추천만 등)
```

**파라미터 활용:**
- `temperature`: 캐릭터 성향에 따라 차등 부여 (예측불가능한 캐릭터 ↑, 일관적 캐릭터 ↓)
- Batch API: 같은 라운드의 병렬 호출을 배치로 묶어 비용 50% 절감
- 모델 선택: 글 작성/핵심 반응은 고성능 모델, 소극적 반응(추천/비추천)은 경량 모델로 차등

### 6.6 3층 기억 구조

```
community_memory.md        ← "다들 아는" 큰 사건/밈/논쟁 (전체 공유)
board_states/[id].md       ← 현재 활성 스레드, 분위기 (게시판별)
persona_states/[id].md     ← 현재 입장/관계/감정 스냅샷 (레귤러 개인별)
                              + 그룹 상태 (미선정 캐릭터 아키타입별)
```

갱신 프로세스: 시뮬레이션 결과 → 오케스트레이터 갱신안 제안 → 작가 승인 → 파일 반영

### 6.7 Layer B (커뮤니티 문화) 관리

**새 커뮤니티/게시판 생성 프로세스:**
```
1. 브레인스토밍: 작가 + 오케스트레이터
   참조 소재: 작가가 직접 제공 / 외부 서치 / 오케스트레이터 추론
2. 정제: LLM이 시뮬레이션 필수 요소 확인 (빠진 것만 질문)
   · 전반적 톤/분위기
   · 주요 이용자 층 (어떤 성향의 캐릭터가 모이는가)
   · 주력 콘텐츠 유형 (논쟁/정보/감정토로/거래 등)
   · 익명 문화 수준
   · 금기/불문율
3. 합의 → layer_b/[커뮤니티명].md 생성
```

**문화 진화**: 시뮬레이션에서 밈/사건 자연 발생 → 오케스트레이터가 Phase 2 상태 갱신 시 Layer B 업데이트 제안 → 작가 승인

## 7. 집필 시스템

### 7.1 집필 5-Phase 사이클

```
[Phase 1: 기획 대화]
작가 ↔ 오케스트레이터
참조: master_map + narrative_state + 직전 Writer Report
    + 주요 캐릭터 상태 카드 + community_memory
  → 회차 윤곽 합의
  → 오케스트레이터가 writing_prompt + 컨텍스트 패키지 정제
  → 작가 최종 확인

writing_prompt 포함 항목:
  1. 에피소드 플롯 윤곽 (대화 합의 내용)
  2. 관련 세계관 섹션 (서브 에이전트 추출)
  3. 등장 캐릭터 카드 + 상태 카드
  4. style_bible.md
  5. Voice Pack (레귤러 등장인물)
  6. narrative_state 관련 항목
  7. 직전 에피소드 마지막 장면

[Phase 2: 작문]
작가가 writing_prompt를 작문 모델에 수동 제출 → 초안 수령

[Phase 3: 수정]
초안 → 작가 손수 수정 → 수정본 확정

[Phase 4: 스타일 학습]
입력: 초안 + 수정본
  → 스타일 분석 에이전트 before/after 비교
  → 패턴 추출 + 기술적 craft 개선 피드백
  → 작가 승인 후 style_bible 업데이트

[Phase 5: 상태 갱신 합의]
입력: 수정본(확정 원고)만 — 초안 사용 금지
  → 오케스트레이터가 갱신안 제안:
     · narrative_state: 회수된 복선, 진행된 아크
     · persona_state: 관계/감정 변화 캐릭터
     · community_memory: 캐논이 된 공개 사건
  ↕ 작가와 대화 (승인/수정/보완)
  → 합의된 내용으로 각 상태 파일 갱신
```

> **입력 분리 원칙**: Phase 4는 초안+수정본 비교, Phase 5는 수정본(확정 원고)만 사용

### 7.2 style_bible.md (2 트랙)

**Track A: 현재 필체** (서술형)
"이 작가는 X 패턴을 선호한다" — 초안 vs 수정본 비교에서 추출
- 문장 수준: 길이, 종결 방식, 어미 패턴
- 묘사 수준: 감정 밀도, 액션 호흡, 감각 표현 선호
- 구성 수준: 장면 전환, 시점, 대화:서술 비율
- 금기 목록: 이 작품에서 절대 쓰지 않는 표현
- 레퍼런스 발췌: 작가 확정 기준 문단 3~5개

**Track B: 성장 방향** (처방형)
"이 방향으로 개선 중이다" — LLM 피드백 + 작가 승인으로 추가
- 피드백 가능 영역: 문장 경제성, 보여주기 vs 설명하기, 리듬 일관성, 클리셰 탐지
- 피드백 불가 영역: 작가의 감성적 판단, 의도적 비관습 선택, 외부 작가 스타일 강요
- 거부 이유 기록 → 동일 피드백 반복 방지

**성장 타임라인:**
```
1~10화:  style_bible 확립기 (작가 교정 위주)
11~30화: 적용기 (수정량 점차 감소)
30화~:   자동 유지 (작가는 감독 역할 집중)
```

**모델 교체 시 대응:**
신규 출력 vs 기존 확정 원고 비교 → 차이점 보고 → style_bible에 보정 지침 추가

### 7.3 narrative_state.md

작가가 직접 관리하는 서사 의도 문서:
- 활성 복선 목록 (심은 회차 / 회수 예정)
- 캐릭터 아크 진행도
- 전체 구성 위치 (1부 중반 등)
- 다음 3~5화 방향
- 문체/톤 가이드

## 8. 초기 부트스트랩 순서

### 8.1 현재 보유 자산

| 자산 | 상태 | 필요 작업 |
|---|---|---|
| 월드 바이블 | 섹션화 완료 (worldbible_refined_bundle) | 점진적 보완 |
| 기존 캐릭터 카드 162개 | **폐기** | P-* 슬롯으로 대체 |
| P-* 모집단 슬롯 | 미생성 | 빅5 배정 스크립트 + 배치 LLM 실행 |
| population_grammar.md | 미작성 | 신규 작성 |
| community_map | 초안 존재 | 검토 후 확정 |
| layer_b 문서 | 초안 존재 | 검토 후 확정 |
| simulation_playbook | 초안 존재 | 검토 후 확정 |
| master_map | 초안 존재 | P-* 체계 반영 후 갱신 |

### 8.2 부트스트랩 순서

```
[1단계] population_grammar.md 작성
  · 탑별 인구 비율, 배경 유형 분포, 집단 간 긴장 구조 정의
  · 시드 유형별 반응 민감도 맵 (어떤 심리 좌표가 어떤 시드에 반응하는가)

[2단계] P-* 슬롯 생성 + 빅5 배정
  · 파이썬 스크립트: 정규분포 샘플링 → P-0001~P-3600 YAML 생성
  · 학술원 보정 적용 (`docs/design/character_population_methodology.md` 기준)

[3단계] 파생 지표 산출
  · Batch API 병렬 호출: 빅5 → DT/NFC/SM 파생
  · character_index 초기 좌표 레지스트리 구성

[4단계] 핵심 인물 5~7명 선제 조형 (CH-*)
  · 주인공 주변 고정 인물 — 매 시뮬레이션의 중력 역할
  · 온라인 성향 레이어도 이 인물들만 사전 작성

[5단계] 첫 시뮬레이션 시나리오 결정
  · 시나리오 범위에 맞게 world_bible 섹션 로딩 범위 확정
  · community_map + layer_b 최종 확인

[6단계] master_map 갱신 후 첫 시뮬레이션 실행
```

> **핵심 변경**: 캐릭터 준비가 더 이상 병목이 아님. 3600개 슬롯은 스크립트로 수분 내 생성. 인물 조형은 시뮬레이션이 대신한다.

```
  → 결과 수합(오케스트레이션) → Writer Report
(정식 모드: + 상태 갱신 작가 승인)

[집필] — 오케스트레이션 레이어
기획 대화 → writing_prompt 조립 → 수동 제출 → 초안
→ 작가 수정 → 스타일 학습 → 상태 갱신 합의

[캐릭터 성장]
풀 캐릭터 → 시뮬레이션 반복 → 레귤러 선정 → 카드 보강
```
