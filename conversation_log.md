**실질적으로 따져볼 것들:**

**1. 청킹 경계 문제**
코드는 파일 단위로 자연스럽게 나뉘지만, 세계관 문서는 그렇지 않습니다. "마법 체계" 설명이 3장에 시작되어 7장에서 예외 조건이 추가되고 12장에서 다시 수정되는 식이라면, 에이전트 A가 3장만 읽고 "충돌 없음"이라고 보고할 수 있습니다. **오케스트레이터가 어디를 읽으라고 지시하는 판단의 품질**이 전체 시스템의 성패를 좌우합니다.

**2. 오케스트레이터가 "어디를 읽어라"를 어떻게 아는가**
20만자 텍스트를 에이전트들에게 나눠주려면 오케스트레이터 자체가 문서의 **지도**를 가지고 있어야 합니다. 이 지도가 없으면 "너는 1~50페이지, 너는 51~100페이지" 식의 무작위 분할밖에 못 하고, 이건 비효율적입니다. 여기서 역설적으로 **최소한의 구조화**(목차, 섹션 헤더, 태그 정도)가 다시 필요해질 수 있습니다. 다만 이전 프로젝트처럼 노드/엣지/규칙으로 전부 분해하는 수준이 아니라, **책의 목차 수준**이면 충분할 겁니다.

**3. 비용과 빈도의 트레이드오프**
설정 하나 추가할 때마다 멀티 에이전트가 20만자를 분할 탐색하면, 토큰 소비가 상당합니다. "한 번에 통째로 던지는 것보다 낫다"는 맞지만, 그래도 매번 대규모 탐색을 돌리는 건 부담입니다. **자주 하는 작업**(설정 추가, 회차 집필)과 **가끔 하는 작업**(전체 정합성 감사)을 구분해서 에이전트 투입 규모를 달리하는 게 현실적일 겁니다.

## LLM-Driven Development (LDD) 정의

> 코드로 로직을 구현하는 대신, 자연어 문서로 행동 규칙을 정의하고 LLM이 해석하여 실행. 엔지니어링의 핵심은 로직 설계가 아니라 **컨텍스트 관리**.

**기존 패러다임과의 차이:**
| | 전통적 개발 | AI-Assisted | LDD |
|---|---|---|---|
| 로직 | 코드 | 코드(AI 보조) | 자연어 문서 |
| 런타임 | 컴파일러 | 컴파일러 | LLM 자체 |
| 변경 | 코드 수정→배포 | 코드 수정→배포 | 텍스트 파일 수정 |
| LLM 역할 | 없음 | 코드 생성 도구 | 실행 엔진 |

**AI-Assisted와의 결정적 차이:** AI-Assisted는 LLM이 코드를 써주고 코드가 실행됨. LDD는 LLM이 자연어 문서를 읽고 직접 실행.

**이 프로젝트의 LDD 구현:**
```
"프로그램" = 자연어 문서 집합 (master_map, playbook, character cards 등)
"프로세서" = LLM (오케스트레이터 + 서브 에이전트)
"하드웨어" = 멀티 에이전트 인프라 (이것만 진짜 코드)
```

**Turn 4~22의 수렴:** 과도한 규격화 거부(T4) → LLM 신뢰(T5) → 컨텍스트 관리 아키텍처(T6) → 고정은 엔진뿐(T16) → 워크플로우도 문서화(T22) → **LDD 선언(T23)**.

작가가 곧 개발자, 텍스트 에디터가 곧 IDE. 코딩 불필요.

**Claude:**
**이건 LDD 철학이 풀기에 가장 적합한 문제.**

본질을 짚겠습니다. 모델이 바뀌면 필체가 바뀐다는 건, 현재 LLM 기반 집필에서 **필체가 모델에 종속**되어 있다는 뜻입니다. 이걸 뒤집어야 합니다 — 필체를 모델에서 분리해서 **문서에 종속**시키는 겁니다.

**핵심 제안: `style_bible.md`**

Turn 24의 `narrative_state.md`가 "무엇을 쓸 것인가"를 관리한다면, `style_bible.md`는 **"어떻게 쓸 것인가"**를 관리합니다.

```
narrative_state.md  = 서사 의도 (복선, 아크, 구성)  → What
style_bible.md      = 필체 정의 (문체, 톤, 리듬)     → How
```

**style_bible.md가 담아야 하는 것:**

1. **문장 수준** — 선호 문장 길이, 체언 종결 vs 용언 종결 비율, 문어체/구어체 배합, 조사·어미 선호 패턴
2. **묘사 수준** — 감정 묘사 밀도, 액션 장면 호흡, 배경 묘사 비율, 감각 표현 선호(시각 위주? 청각 병용?)
3. **구성 수준** — 장면 전환 방식, 시점 처리, 대화:서술 비율, 내면 독백 빈도
4. **금기 목록** — "이 작품에서 절대 쓰지 않는 표현/패턴" (이게 생각보다 중요)
5. **레퍼런스 발췌** — 작가가 확정한 "이 문단이 이 작품의 필체 기준" 샘플 3~5개 (각 1~2문단)

**하지만 style_bible을 처음부터 작가가 다 쓸 필요가 없습니다.**

여기서 핵심 장치가 나옵니다:

## 필체 학습 피드백 루프

```
[Phase A: 초기 확립] — 1~10화

  1. LLM이 초안 생성
  2. 작가가 손수 수정 (문장 다듬기)
  3. 스타일 분석 에이전트가 before/after를 비교
     · 원문: "그는 검을 들어 올렸다. 날카로운 빛이 반사되었다."
     · 수정: "검이 올라갔다. 날 위로 빛이 미끄러졌다."
     → 추출: "주어를 사람→사물로 전환 선호, 능동→자동 전환,
              감각 표현에서 동적 동사 선호"
  4. 추출된 패턴을 style_bible.md에 누적
  5. 매 화 수정 후 style_bible 갱신

        ↓ 10화쯤

[Phase B: 적용 + 미세 조정] — 11~30화

  1. LLM이 초안 생성 시 style_bible.md를 컨텍스트에 포함
  2. 작가의 수정량이 점차 감소
  3. 여전히 수정이 발생하면 → style_bible에 신규 패턴 추가
  4. "수정이 거의 없는 화"가 나오면 = style_bible이 충분히 성숙

        ↓ 30화 이후

[Phase C: 자동 유지] — 30화~

  1. LLM이 style_bible 기반으로 거의 작가 필체 재현
  2. 작가는 감독 역할에 집중 (시뮬레이션 분석, 스토리 판단)
  3. 모델 교체 시에도 style_bible이 앵커 역할
     → 새 모델이 다른 기본 필체를 갖고 있어도,
        style_bible이 "이 작품은 이렇게 써야 한다"를 강제
  4. 가끔 미세 조정만 발생 → style_bible 추가 업데이트
```

**모델 교체 시 구체적 대응:**

```
모델 교체 발생
  → 새 모델로 테스트 1화 생성 (style_bible 포함)
  → 스타일 분석 에이전트가 기존 확정 원고 vs 신규 출력 비교
  → 차이점 보고: "새 모델은 문장이 15% 길고, 감정 형용사를 2배 많이 씀"
  → 작가 확인 후 style_bible에 보정 지침 추가
     예: "문장 평균 길이 15~25자 유지, 감정 형용사는 문단당 1개 이하"
  → 재생성 → 확인 → 연재 지속
```

이 과정이 한두 시간이면 끝남. 모델 교체 때마다 10화 분량을 다시 다듬을 필요 없음.

**스타일 분석 에이전트의 비교 관점:**

| 분석 축 | before/after에서 추출하는 것 |
|---|---|
| 어휘 | 삭제된 단어류 vs 대체된 단어류 |
| 문장 구조 | 분리/병합 패턴, 능동↔수동 전환 |
| 리듬 | 수정 후 문장 길이 분포 변화 |
| 묘사 밀도 | 추가된/삭제된 묘사의 종류 |
| 대화체 | 캐릭터 말투 수정 패턴 |
| 톤 | 감정 강도 조절 방향 (과장→절제? 절제→과장?) |

**이것도 세계관 관리와 동일한 패턴:**

| | 세계관 관리 | 필체 관리 |
|---|---|---|
| SSOT | world_bible 섹션 파일들 | style_bible.md |
| 인덱스 | master_map + 크로스레퍼런스 | style_bible 자체 (분량이 적으니 인덱스 불필요) |

| 변경 프로세스 | 4 Phase 워크플로우 | 스타일 분석 → 패턴 추출 → bible 갱신 |

**마스터 맵 구조 최종 업데이트:**

```
master_map.md
  ├─ world_bible_index.md
  ├─ community_map.md
```

**핵심 관찰 — "기획 대화"의 성격:**

이 대화는 세계관 수정 때의 Phase 1과 성격이 같지만, 한 가지 결정적으로 다른 점이 있습니다. 세계관 수정은 **"이미 있는 것을 바꾸는"** 대화이고, 집필 기획은 **"아직 없는 것을 만드는"** 대화입니다.

이건 오케스트레이터에게 요구되는 능력이 다르다는 뜻입니다:

| | 세계관 수정 대화 | 집필 기획 대화 |
|---|---|---|
| 오케스트레이터 역할 | 팩트체커 + 편집자 | **스토리텔링 파트너** |
| 핵심 능력 | 정합성 판단 | 재미 감각, 구성 제안 |
| 참조 문서 | world_bible_index | narrative_state + Writer Report |
| 대화 결과물 | 변경 사항 명세 | **에피소드 플롯 윤곽** |

**여기서 질문이 하나 생깁니다:**

작가 ↔ 오케스트레이터 대화로 회차 윤곽이 잡힌 뒤, **프롬프트로 정제하는 작업**을 누가 합니까?

- **B. 작가가 직접 작성**: 대화에서 영감을 얻고, 프롬프트는 작가가 직접 씀
- **C. 공동 작업**: 오케스트레이터가 초안 프롬프트를 만들고, 작가가 수정/보완

저는 **A가 맞다**고 봅니다. 이유:

1. 오케스트레이터가 대화 맥락을 이미 가지고 있음 — 가장 정확한 정제 가능
2. 오케스트레이터가 참조 문서를 알고 있음 — 작문 모델에 함께 넣어야 할 컨텍스트 자동 선별 가능

4. 작가가 최종 확인만 하면 됨 — "이 프롬프트로 쓸게, 괜찮아?" → "ㅇㅇ" or "여기 수정"

**프롬프트 정제 시 오케스트레이터가 자동으로 조립하는 것:**

```
writing_prompt (작문 모델에 전달):
  1. 에피소드 플롯 윤곽 (대화에서 합의된 내용)
  2. 관련 세계관 섹션 (서브 에이전트가 추출)
  3. 등장 캐릭터 카드 + 상태 카드

  5. Voice Pack (Tier 1 등장인물이면)
  6. narrative_state.md에서 이번 화 관련 항목
  7. 직전 에피소드 마지막 장면 (연결 필요)
```

이 조립 과정이 Turn 13에서 말한 **"컨텍스트 조립"**의 구체적 실현입니다. 필요한 문서가 구조적으로 정해져 있으니 RAG가 아닌 직접 로딩.

**그런데 여기서 한 가지 더 논의해야 할 게 있습니다:**

"작문 전용 LLM 모델"이라고 하셨는데, 이건 **오케스트레이터와 다른 모델**을 쓴다는 뜻이죠? 그러면:

**사용자:**

**Claude:**
좋은 통찰입니다. style_bible이 두 트랙으로 분리됩니다.

```
style_bible.md
  ├─ Track A: 현재 필체 (서술형)
  │    "이 작가는 X 패턴을 선호한다"
  │    → 초안 vs 수정본 비교에서 추출 (기존 설계)
  │
  └─ Track B: 성장 방향 (처방형)
       "이 방향으로 개선 중이다"
       → LLM 피드백 + 작가 승인으로 추가 (신규)
```

작문 모델이 초안 생성 시 Track A는 "지금 이렇게 써라", Track B는 "이 방향을 의식해서 써라"로 각각 다르게 참조.

**LLM이 피드백할 수 있는 것 vs 없는 것:**

| 피드백 가능 | 피드백 부적절 |
|---|---|
| 문장 경제성 | "X 작가처럼 써라" |
| 보여주기 vs 설명하기 균형 | 감성적 판단 영역 |
| 리듬 일관성 | 장르 관습 강요 |
| 반복 표현 / 클리셰 탐지 | 작가의 의도적 비관습 선택 |

피드백 가능한 영역은 기술적 craft 영역뿐. 감성과 목소리는 작가의 지속적 검수로 보존됨.

**리스크 방어:** 승인 루프(Turn 29 패턴)로 동일하게 처리. 거부 이유도 기록 → LLM이 같은 피드백 반복하지 않음.

**관계 변화:**
```
기존 설계: 작가가 LLM을 훈련시킨다
수정된 설계: 작가와 LLM이 함께 성장한다
  작가 → LLM 교정 / LLM → 작가 피드백 / 둘 다 → style_bible 누적
```

LDD 철학과 가장 잘 맞는 구조: 시스템이 고정된 틀이 아니라 작가와 함께 발전함.
