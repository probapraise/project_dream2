## Board Lifecycle

### 1. `concept_only`
- 장면 기획/시뮬레이션 시드 안에서만 쓰는 임시 보드 후보
- 아직 recurring surface가 아니므로 `BOARD-###` ID를 부여하지 않는다.
- `community_map.md` Active communities에 등록하지 않는다.
- run 문서나 프롬프트 안에서는 기능 설명형 이름으로만 부른다.

### 2. `registered`
- 작가가 반복 사용 가능한 게시판으로 승인한 상태
- 이 단계에서만 새 `BOARD-###` ID를 부여한다.
- `community_map.md` Active communities에 등록한다.
- 아직 지속 상태가 필요 없다면 `board_states/` 파일은 만들지 않는다.

### 3. `stateful`
- 밈, 운영 이슈, ongoing conflict, 누적된 판례처럼 다음 장면에도 남겨야 할 상태가 생긴 보드
- 이 단계부터 `board_states/`에 요약 상태 파일 또는 승인된 상태 스냅샷을 둔다.
- 같은 보드의 run 결과가 여러 번 나오더라도, live에 남길 정보는 승인 후 선별 반영한다.

### 4. `retired`
- 더 이상 활성 구조에 쓰지 않는 상태
- Active communities에서 제거한다.
- 필요 시 history/quarantine에만 흔적을 남긴다.

# Community Grammar — Layer B

## 정의

레이어 B는 **작가 투사 레이어**다.
주인공(빙의 개발자)이 전생에서 사랑했던 인터넷 커뮤니티 문법을 각인광장 안에 의도적으로 이식하는 계층.

실제 커뮤니티의 글·사건을 분해 → ATOM으로 저장 → 일정량 누적 시 상위 GRAMMAR 규칙으로 합성하는 파이프라인으로 운영한다.

---

## 적용 범위

### 결정 (2026-03-04)

레이어 B는 **각인광장 내 주인공이 직접 관리하는 단일 익명 게시판 "낙서장"**에만 적용한다.

#### 성격

| 항목 | 내용 |
|---|---|
| 소유/관리 | 주인공이 가장 아끼고 직접 운영 |
| 목적 | 순수 취미 — 주인공 개인 취향에 의해서만 유지 |
| 상업성 | 없음. 각인광장의 상업적 성장·LMM 고도화 등 외적 목표와 **의도적으로 배치** |
| 익명성 | 익명 게시판 |
| 서사적 기능 | 주인공의 고집과 정체성을 드러내는 핵심 장치 |

#### 내러티브 마찰

- 작품이 진행될수록 주인공 주변의 조력자들은 이 게시판의 존재를 문제시한다.
- "제발 저 문제 많은 익명 게시판 좀 없애고 쉽게 좀 가자"는 하소연이 반복된다.
- 그럼에도 주인공은 포기하지 못한다 — 이 공간은 주인공이 전생에서 그리워하는 커뮤니티 경험의 재현이기 때문.
- 이 갈등은 주인공의 캐릭터 아크에서 **합리적 경영자 vs. 감성적 커뮤니티 덕후** 사이의 긴장으로 기능한다.

#### 적용 범위가 단일 게시판인 이유

- 레이어 B의 문법은 특유의 익명성·혼돈·내부자 유머에 기반하므로, 각인광장 전체에 깔면 톤이 오염된다.
- 국소적 적용이 서사적 마찰(조력자들의 반대)을 자연스럽게 만든다.
- 주인공의 집착 대상이 명확해야 감정선이 선명하다.

## ATOM 저장 형식

```yaml
ATOM-NNN:
  source_type: 게시글 / 댓글 / 밈 패턴 / 사건
  core_pattern: 추상 사회 동학
  emotional_register: 감정 온도 (냉소적 쾌감 / 훈훈한 연대 / 집단 조롱 등)
  in_group_mechanics: 집단 내 작동 원리
  trigger_condition: 발동 조건 (캐릭터 타입 + 상황)
  world_translation: 각인광장 표면형 (세계관 언어 번역)
```

# population_grammar

작성일: 2026-03-04
버전: v0.3 (dorm-rule sync)

## 1) 목적
- `P-*` 모집단(총 3,600명)의 기본 분포 문법을 고정한다.
- 오케스트레이터가 시드에 맞는 반응자 후보를 빠르게 추론할 때 기준으로 사용한다.

## 4) 배경 유형 분포 (초기 운영 priors)
배경 유형은 캐릭터 조형 시 "서사적 출신 축"으로 사용한다.

정의:
- 서명귀족: 문장비전 혈통키 보유 계열(비전관 축과 강결합)
- 일반귀족: 문장비전 미계승자/분가귀족/기사귀족 포함(학술원 다수축)
- 평민: 도시 시민/농민/하급 기능직. 운영 통계상 몰락귀족은 평민에 편입
- 외국인: 타국 조약/파견/교환학생 인간권
- 인외: 엘프/드워프/수인/고블린/리자드포크 등 비인간 계열

슬롯 코드 매핑(`background_type`):
- `signature_noble` = 서명귀족
- `common_noble` = 일반귀족
- `commoner` = 평민(몰락귀족 편입)
- `foreigner` = 외국인
- `nonhuman` = 인외

보유 플래그 매핑(`crest_vision_holder`, bool):
- `true` = 문장비전 보유자
- `false` = 문장비전 비보유자
- v1 부트스트랩에서는 기본값을 `signature_noble=true`, 그 외 `false`로 시드화하되, 필드는 `background_type`과 독립 변수로 유지한다.

초기 배정안(합 3,600):

| 배경 유형 | 인원 | 비율 | 근거 메모 |
|---|---:|---:|---|
| 서명귀족 | 160 | 4.4% | 24가문×학년별 1~2명 계승자 가정의 운영 기준치(`WB-0015` 13.3.5) |
| 일반귀족 | 2,870 | 79.7% | 미계승/분가/기사귀족 다수층(`WB-0026` 0-C, 0-E) |
| 평민 | 200 | 5.6% | 서명귀족보다 약간 큰 축(몰락귀족 포함 운영) |
| 외국인 | 120 | 3.3% | 타국 조약/파견 120명 운영축(`WB-0015` 13.6 종족표) |
| 인외 | 250 | 6.9% | 종족 분포상 비인간 250명(`WB-0015` 13.3.4) |
| 합계 | 3,600 | 100% |  |

운영 메모:
- 배경 유형은 통계적 우선값이다. 개별 인물 조형 시 예외 허용.
- 평민 중에서도 2~3세대 상위 선조에 귀족 계통이 있는 사례를 흔한 값으로 가정한다.
- 외국인(인간)과 인외는 중복 분류하지 않는다. 1인 1배경 유형 원칙.
- 본 버전의 운영 priors에서는 문장비전 보유자 총량을 160명으로 고정한다(출신 유형과 독립).
- 외국인/인외는 기숙사 배정에서 동일 규칙을 적용한다(문장비전 보유 여부 + 직능 기준).

## 5) 기숙사 배정 규칙 (최신 합의)
본 장은 "우선 경향"이 아니라 운영 강제 규칙을 포함한 SSOT로 적용한다.

### 5.1 배정 우선순위
1. 문장비전 보유자(`crest_vision_holder=true`)는 배경 유형과 무관하게 비전관 강제 소속.
2. 문장비전 비보유자는 직능(`vocation`) 기준으로 기본 기숙사 배정:
   - 7탑 기숙사군: 마도/연구/실습 중심 직능
   - 기사동: 기사/경비/전술 중심 직능
   - 신전동군: 치유/의례/성직 중심 직능
3. 외국인/인외는 동일 기준 배정하며 별도 전용동을 두지 않는다(외국·인외동 폐지).
4. 정원 초과 시 동일 직능군 내 분반을 우선 적용하고, 임시 전환 배정은 다음 라운드에서 재조정한다.

### 5.2 배경 유형별 적용표
| 배경 유형 | 문장비전 보유 시 | 문장비전 비보유 시 | 메모 |
|---|---|---|---|
| 서명귀족 | 비전관(강제) | 직능별 기본 기숙사 | 보유율이 가장 높지만 판정 기준은 "보유 여부" 단일 |
| 일반귀족 | 비전관(강제) | 직능별 기본 기숙사 | 미계승/분가 포함 동일 규칙 |
| 평민 | 비전관(강제) | 직능별 기본 기숙사 | 몰락귀족 편입 운영 유지 |
| 외국인 | 비전관(강제) | 직능별 기본 기숙사 | 인외와 동일 규칙 |
| 인외 | 비전관(강제) | 직능별 기본 기숙사 | 외국인과 동일 규칙 |

### 5.3 기숙사 분포 예시 (합 3,600 / 보유자 160 가정)

## 7) 시드 유형별 반응 민감도 맵 (빅5 좌표 구간)

구간 정의:
- Low: `0.01-0.35`
- Mid: `0.36-0.64`
- High: `0.65-0.99`

### 7.1 정렬/서열 조작 의혹 (정렬/공정성 논쟁 중심)
- 고반응 후보:
  - `C High` + `N High`: 점수/절차 불공정에 예민, 장문 근거 제시
  - `E High` + `A Low`: 공개 비난/선동형 반응
- 저반응 후보:
  - `E Low` + `N Low`: 관망, 확증 전 발화 회피

### 7.2 봉문/검열 스캔들 (공보국/운영 이슈)
- 고반응 후보:
  - `O High` + `A Low`: 표현 통제에 강한 반발, 우회담론 생산
  - `E High` + `N High`: 급속 확산형 감정 동원
- 저반응 후보:
  - `C High` + `A High`: 규정 우선, 신중/완곡 표현

### 7.3 저주/오염/안전사고 이슈 (안전/구호/루머 확산 중심)
- 고반응 후보:
  - `N High` + `A High`: 공포 + 보호본능, 구호/규제 지지
  - `N High` + `A Low`: 책임 추궁/희생양 탐색 강화
- 저반응 후보:
  - `N Low` + `E Low`: 사실 확인 전 거리두기

### 7.4 후원/특혜/추천권 거래 의혹
- 고반응 후보:
  - `A Low` + `N High`: 특권 비판, 공격적 프레임 경쟁
  - `C High` + `O Mid~High`: 제도 개편형 장문 주장
- 저반응 후보:
  - `A High` + `E Low`: 갈등 회피, 중립 발화

### 7.5 문화충돌/예절 논쟁 (외국인·인외 통합배정 이슈)
- 고반응 후보:
  - `O High` + `A High`: 중재/번역/맥락화 시도
  - `E High` + `A Low`: 조롱/도발/멸칭 확산 위험
- 저반응 후보:
  - `O Low` + `E Low`: 타문화 맥락 해석 포기

### 7.6 직능기숙사 통금/생활규정 강화
- 고반응 후보:
  - `C Low` + `E High`: 규율 반발, 집단 행동 촉발
  - `C High` + `N High`: 불안 기반 규정 재설계 요구
- 저반응 후보:
  - `C High` + `A High`: 순응적 반응

## 8) 오케스트레이터 사용 규칙
1. 시드 분류(검열/서열/안전/특혜/문화충돌 등) 후 7장 민감도 맵에서 고반응 좌표를 선택한다.
2. 3장 분포를 벗어나지 않는 범위에서 후보를 샘플링한다.
3. 6장 긴장축을 우선 적용해 반응 강도(무반응/소극/댓글/핵심)를 나눈다.
4. 과포화 방지: 동일 축(`탑 x 학년 x 배경`) 집중도가 25%를 넘으면 다른 축으로 분산한다.

## 9) EX-* 외곽 인물 분류

### 9.1 정의
- `EX-*`는 학술원 권역 바깥의 서사 인물(가족, 귀족, 상인, 타국 인물 등)을 관리하는 슬롯이다.
- 저장 경로: `external/EX-*.yaml`

### 9.2 P-*/NS-*와의 관계

| 분류 | 범위 | 시뮬 필드(빅5/파생) | 커뮤니티 시뮬 풀 |
|---|---|---|---|
| P-* | 학술원 학생 | O | O (각인광장) |
| NS-* | 학술원 내 비학생 | O | O (교수/직원 등) |
| EX-* | 학술원 외곽 | O (미래 대비) | X (대기) |

### 9.3 YAML 스키마

```yaml
id: EX-NNNN
name: (확정 시 기입, 미확정 시 null)
role: (서사 역할 — 예: 렌바렌 백작가 가주, 상단 대표 등)
affiliation: (소속 조직/가문/국가)
b5:
  O: (0.01-0.99)
  C: (0.01-0.99)
  E: (0.01-0.99)
  A: (0.01-0.99)
  N: (0.01-0.99)
derived:
  DT: (0.01-0.99)
  NFC: (0.01-0.99)
  SM: (0.01-0.99)
status: uninstantiated | instantiated | named
ch_id: null | (확정 시 부여)
# --- EX-* 전용 필드 ---
narrative_fields:
  goals: (캐릭터의 목표/욕망)
  secrets: (비밀 — 정보 레벨 표시)
  core_relationships: (핵심 관계 목록)
  arc_notes: (캐릭터 아크 메모)
community_eligible: false  # true 전환 시 NS-* 승격 또는 별도 분류 신설
```

### 9.4 운영 원칙
- EX-*는 빅5/파생지표를 처음부터 보유하되, 커뮤니티 시뮬레이션 풀에는 투입하지 않는다.
- `narrative_fields`는 EX-* 전용이며, P-*/NS-*에는 존재하지 않는 서사 카드 필드이다.
- 작품 진행에 따라 커뮤니티 참여 가능성이 생기면 `community_eligible: true`로 전환 후 NS-*로 승격하거나 새 분류를 신설한다.
- EX-* 슬롯 번호는 0001부터 순차 부여한다.
