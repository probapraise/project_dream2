# Root Solution: 데이터 손실 없는 충돌 정리

## 문제 원인
- 파일 단위 삭제를 먼저 수행하면, 충돌 구간뿐 아니라 유효한 설정까지 함께 소실된다.
- 인덱스 재생성만으로는 손실된 설정 맥락이 복구되지 않는다.

## 해결 원칙
1. **통삭제 금지 기본값**: 기본 전략은 부분 수정/격리다.
2. **라인 단위 충돌 처리**: 충돌은 파일이 아니라 "구간(line span)" 단위로 식별한다.
3. **살려야 할 내용 우선 추출**: 비충돌 구간을 먼저 유지/이관한 뒤 충돌 구간만 제거한다.
4. **삭제는 예외 경로**: 파일 전체가 충돌일 때만 삭제를 허용한다.
5. **항상 복구 가능 상태 유지**: 삭제 전 원본 복구 기준(commit)과 복구 명령을 세션 문서에 기록한다.

## 표준 절차
### Step 1) 충돌 맵 작성
- 세션 문서에 파일별 충돌 구간을 기록한다.
- 형식: `path + line start/end + 충돌 근거 + 처리안`

### Step 2) 처리안 분류
- `KEEP`: 그대로 유지
- `REWRITE`: 구간 수정
- `QUARANTINE`: 작중 비노출/보조 문서로 격리
- `DROP`: 구간 삭제

### Step 3) 삭제 게이트
- 파일 삭제가 필요하면 `deletion_gate` 문서를 필수 작성한다.
- 필수 조건:
  - 파일 전체가 충돌임을 증명
  - 비충돌 정보가 없음을 명시
  - 복구 기준 commit 기록
  - 작가 승인 기록

### Step 4) 반영 후 감사
- `scripts/world_ops_audit_bundle.sh` 실행
- 세션/로그 갱신

## 삭제 예외 허용 조건
- 아래 4개를 모두 만족할 때만 파일 삭제 허용:
  1. 충돌 구간 비율 100%
  2. 유지할 유효 정보 없음
  3. 복구 기준 commit 기록 완료
  4. 작가 승인 완료

## 운영 메모
- 빠르게 지우는 것보다, 나중에 되돌릴 필요가 없는 정리가 우선이다.
- 충돌 가능성이 높아도 "데이터 보존"이 기본값이다.
